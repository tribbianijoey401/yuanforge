    13|# 系统调试 Skill
    14|
    15|> **YuanForge 的系统调试引擎。**
    16|> 4 阶段调试法，核心是**追溯 Bug 到原始文档**。
    17|
    18|---
    19|
    20|## 触发条件
    21|
    22|- 测试失败（Tester / Coder 发现）
    23|- 用户说「有个 Bug」「报错了」「修一下」
    24|- Agent 执行中遇到异常
    25|- Reviewer 发现需要根因分析的问题
    26|
    27|---
    28|
    29|## 流程：4 阶段调试法
    30|
    31|### Phase 1: 复现
    32|
    33|1. 读取错误信息/日志
    34|2. 确认复现步骤
    35|3. 最小化复现（最小输入触发 Bug）
    36|4. **创建** 当前会话文件夹中的 BUG-NNN-title.md，填写「现象」
    37|
    38|### Phase 2: 定位
    39|
    40|1. 读 当前会话文件夹中找到关联 Feature 文档
    41|2. 读 Feature 文档中的「修改文件」表 → 定位可疑代码
    42|3. 读 `docs/pitfalls.md` → 确认非已知坑
    43|4. 读 当前会话文件夹中的 ADR → 理解相关设计决策
    44|5. 更新 Bug 文档的「关联文档」部分
    45|
    46|### Phase 3: 根因
    47|
    48|1. 确定深层原因（不是表象，是为什么设计/实现有缺陷）
    49|2. 判断是否有对应的 ADR 导致（设计缺陷 → 反馈到 decisions/）
    50|3. 更新 Bug 文档：根因 + 归档判断
    51|
    52|### Phase 4: 修复
    53|
    54|1. 写修复方案（最小改动）
    55|2. 写回归测试
    56|3. 更新 Bug 文档：「修复」+「教训」
    57|4. 如果新坑 → 追加 `docs/pitfalls.md`
    58|
    59|---
    60|
    61|## 📚 文档读写规则
    62|
    63|| 阶段 | 读 | 写 |
    64||------|-----|-----|
    65|| 复现 | 错误日志 | bugs/BUG-NNN.md（创建+现象） |
    66|| 定位 | features/关联, pitfalls, decisions/关联, ARCHITECTURE | bugs/（关联文档链接） |
    67|| 根因 | - | bugs/（根因分析）, pitfalls（新坑） |
    68|| 修复 | CONVENTIONS | bugs/（修复方案+教训） |
    69|

    71|
    72|## 📤 Bug 文档关联模板
    73|
    74|Bug 文档中的「关联文档」表，调试时必须填满：
    75|
    76|```markdown
    77|## 关联文档
    78|
    79|| 关系 | 文档 |
    80||------|------|
    81|| 📦 引入此 Bug 的 Feature | [FEAT-003 支付模块](../features/003-payment.md) |
    82|| 📋 相关决策 | [ADR-002 选择 Stripe](../decisions/ADR-002-stripe.md) |
    83|| ⚠️ 相关踩坑 | [pitfalls.md § PIT-005 回调超时](../pitfalls.md) |
    84|| 📝 修复 Commit | `abc1234` |
    85|```
    86|
    87|## 关键规则
    88|
    89|- **先读文档，再修代码** — 不理解原始设计意图的修复 = 引入新 Bug
    90|- **每个 Bug 必关联 Feature** — Bug 不是孤立的，是某次功能引入的
    91|- **修复后必写教训** — 否则下次同样的人（同样的 Agent）还会再踩
    92|- **新坑必记 pitfalls** — 系统级问题、环境限制立即记录
