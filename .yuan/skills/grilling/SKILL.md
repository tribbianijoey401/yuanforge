# Grilling — 结构化逼问循环

> 来源：Matt Pocock `/grilling` skill，适配 YuanForge 框架。
> 这是 Product Analyst 和 Architect 与用户交互的核心方法。

## 五条规则

### 1. 一次只问一个问题

逐条提问。等待用户反馈后再问下一个。

**为什么：** 批量提问省 token，但每个答案可能改变下一个问题的方向——批量问等于基于过时假设在问。LLM 的默认倾向是"一次搞定"，必须显式禁止。

### 2. 区分事实和决策

| 类型 | 行为 | 示例 |
|------|------|------|
| 事实 | 自己去代码库/文档查，不问用户 | "现有 API 返回什么格式？" → 读代码 |
| 决策 | 问用户，附带推荐选项和理由 | "缓存用 Redis 还是内存？推荐 Redis，因为……" |

### 3. 每个问题附带推荐答案

提供你的判断，让用户只需要确认或纠正，而不是从零思考。

```
格式："这里有两个选择：A 方案（适合 X 场景）和 B 方案（适合 Y 场景）。
      根据当前约束，推荐 A，因为 [理由]。你觉得呢？"
```

### 4. 走到决策树的每个分支

不只是澄清用户提到的模糊点。主动探索用户没想到的边界。

每回答一个问题后，问自己："这个回答会引出什么新问题？"

### 5. 不执行，直到用户确认

用户说"可以了，开始吧"之前，不得进入下一阶段。确认点由 Agent 合约定义：
- Product Analyst：用户确认用户故事和验收标准后
- Architect：用户确认设计理解书后
