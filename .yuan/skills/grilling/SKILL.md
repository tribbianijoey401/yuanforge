# Grilling — 结构化逼问循环

> 来源：Matt Pocock `/grilling` skill 适配 YuanForge；5 维度框架吸收自需求澄清方法论。
> 本 Skill 解决两件事：① 怎么问（对话机制）；② 按哪 5 维度问（覆盖完整性门禁）。

## 第一性原理

需求→可执行规格存在「认知差」：人类需求含隐含假设，AI 无法读取。消除靠系统性提问。
提问必须结构化（LLM 默认「一次搞定」会基于过时假设问）+ 覆盖完备（盲区即 bug）。

5 维度从「业务系统需求→代码」必须消除的认知盲区类别推导，非行业惯例：
- 范围（做/不做）、交互（怎么操作）、异常（出错怎么办）、数据（字段长啥样）、非功能（性能/安全约束）
任一维度缺失 → 该维度代码基于假设生成 → 返工。故 5 维度是过程门禁，不是输出格式。

## 一、对话机制（怎么问）

规则 1：一次只问一个。逐条提问，等反馈再问下一个。批量问=基于过时假设在问。
规则 2：区分事实与决策。
| 类型 | 行为 | 示例 |
|------|------|------|
| 事实 | 自己去代码库/文档查，不问用户 | "现有 API 返回什么格式？" → 读代码 |
| 决策 | 问用户，附带推荐选项和理由 | "缓存用 Redis 还是内存？推荐 Redis，因为……" |
规则 3：每问题带推荐答案。让用户只确认/纠正，不从零想。
```
格式："这里有两个选择：A 方案（适合 X 场景）和 B 方案（适合 Y 场景）。
      根据当前约束，推荐 A，因为 [理由]。你觉得呢？"
```
规则 4：走到决策树的每个分支。主动探索用户没想到的边界；每答完自问「这引出什么新问题」。
规则 5：不执行直到确认。用户说「可以了」前不进下一阶段。确认点由 Agent 合约定义：
- Product Analyst：用户确认用户故事和验收标准（+5 维度澄清记录）后
- Architect：用户确认设计理解书后

## 二、维度覆盖门禁（按哪 5 维度问）

> PA 必须证明覆盖以下 5 维度，否则不得进入「产出用户故事+AC」阶段。维度是提问过程约束，不是独立文档。

维度 1 范围边界 — 盲区：本期做啥/不做啥/依赖/分期
  检查点：MVP 明确、本期不做已文档化、依赖识别
维度 2 交互流程 — 盲区：入口/操作路径/成功失败反馈/空状态
  检查点：每功能入口、空状态、加载态、导航跳转
维度 3 异常边界 — 盲区：PRD 没提但必遇的异常
  检查点：空值/长度/特殊字符、网络失败、并发、权限不足
维度 4 数据规则 — 盲区：字段类型/长度/必填/校验/默认值
  检查点：每字段类型、必填、校验规则、枚举、错误提示
维度 5 非功能需求 — 盲区：性能/兼容/安全/可维护
  检查点：分页、响应时间、浏览器兼容、脱敏、审计、限流

维度剪裁：
- 维度与项目无关 → 显式跳过（如纯后端无 UI → 交互维度减；内部工具 → 非功能可简），跳过须声明理由并记入澄清记录
- 维度内 PRD 已清楚 → 只问 1-2 个确认；不清楚 → 2-4 个

异常处理：
- 用户答「不确定/没想好」→ 记「待确认」，建议后续与业务方确认，不阻塞
- 用户答与已知信息矛盾 → 指出矛盾请确认

## 三、产出吸收（单文档真相源）

5 维度澄清不产独立文档，吸收进 PA 产出物：
- 用户故事：承载维度 1（范围）
- 验收标准(AC)：按维度 2/3/4 组织为 Given/When/Then + 数据规则表 + 异常表
- 风险标签：维度 5 中的安全/性能映射 R0/R1/R2
- 澄清 Q&A 作为 AC 文档内「澄清记录」附录留存溯源（满足铁律Ⅵ 文档即代码）

禁止：同时产自由格式澄清文档 + 用户故事+AC（双真相源→漂移，违反铁律Ⅵ）。
