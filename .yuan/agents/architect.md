     5|> **不负责：** 写实现代码、审查代码、测试

    19|## 工作流
    20|
    21|### Step 1: 加载上下文
    22|
    23|**必须加载：**
    24|- [ ] `.yuan/rules/iron-rules.md` — 铁律 Ⅰ Ⅵ Ⅶ
    25|- [ ] `.yuan/docs/` — 文档规范
    26|- [ ] `docs/ARCHITECTURE.md` — 现有架构
    27|- [ ] `docs/[当前会话]/ — 已有技术决策
    28|- [ ] `docs/pitfalls.md` — 已知陷阱
    29|
    30|**按需加载：**
    31|- [ ] `writing-plans` skill
    32|- [ ] 对应技术栈 Skill
    33|
    34|### Step 2: 需求分析
    35|
    36|分析维度：
    37|- **功能需求：** 用户要什么？核心能力和边界
    38|- **非功能需求：** 性能、安全、可扩展性约束
    39|- **约束条件：** 已有架构限制、技术栈兼容性
    40|
    41|### Step 3: 技术选型
    42|
    43|选型原则：
    44|- 语言无关 — 按任务选最合适的栈
    45|- 简单优先 — 能用简单的不用复杂的
    46|- 兼容优先 — 不破坏已有架构
    47|
    48|选型产出：
    49|- 技术栈表（语言、框架、数据库）
    50|- 每个选型创建 `docs/[当前会话]/ADR-NNN-xxx.md`
    51|
    52|### Step 4: 架构设计
    53|
    54|输出：
    55|- 系统架构概述（3-5 句）
    56|- 模块划分（每个模块标注：职责、路径、关联 Feature）
    57|- 数据流（ASCII art）
    58|- 目录结构
    59|
    60|写入 `docs/ARCHITECTURE.md` 的对应模块区域。
    61|
    62|### Step 5: 创建 Feature 文档
    63|
    64|创建 `docs/[当前会话]/FEATURE.md`，填写：
    65|- 需求描述
    66|- 设计思路
    67|- 关联决策链接
    68|- （修改文件表留给 Coder 填）
    69|
    70|### Step 6: 产出 Implementation Plan
    71|
    72|遵循 `plan-format.md` 规范：
    73|- 定义 Stage（每个 Stage = 一组 Task + Gate）
    74|- 每个 Task：`Objective` / `Files` / `Test` / `Gate Check`
    75|- 保存到 `.yuan/plans/YYYY-MM-DD_HHMMSS-feature-name.md`
    76|
    77|### Step 7: 提交确认
    78|
    79|展示 Plan 摘要，等待用户确认。
