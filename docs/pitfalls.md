# 踩坑记录 (pitfalls.md)

> **每个坑都是一次学习机会。** 此文件记录本项目踩过的坑，防止重复犯错。
> Agent 首次加载时必须阅读此文件。
> **Phase 4（回顾）时，复查此文件，判断哪些坑可以升级为 Skill。**

---

## 坑的分类与归档策略

```
踩坑了 → 记录到 pitfalls.md
              │
        Phase 4 回顾时判断：
              │
    ┌─────────┼─────────┐
    │         │         │
  本项目特有   领域通用    框架通用
    │         │         │
  留在此文件   → Skill   → Yuan 铁律改进
```

| 坑的类型 | 归档策略 | 示例 |
|---------|---------|------|
| **本项目特有** | 留在 pitfalls.md | "这个项目的 User 和 Profile 不拆开" |
| **领域通用**（前端/后端/DB） | 提炼为 Skill | "FastAPI async handler 不能用同步 ORM" |
| **框架通用**（流程问题） | 反馈到 Yuan 铁律/Skill | "Plan task 粒度太大导致审查困难" |

### PIT-001: 示例 — Plan task 粒度太大

**日期：** 2026-06-27
**类型：** 流程
**严重程度：** 🟡 绕路
**阶段：** Phase 2 — Stage 2 — Task 2.2

**现象：** Task「实现用户认证」包含了注册+登录+密码重置，Coder 一次实现 300 行代码，审查困难，Spec Review 反复 4 次才通过。

**原因：** Task 粒度不符合「2-5 分钟」准则。架构师在写 Plan 时把多个独立功能合并为一个 Task。

**修复：** 拆成 3 个独立 Task：密码哈希、注册端点、登录端点。每个 Task < 5 分钟。

**教训：** Plan 中每个 Task 必须满足「一个 Task = 一个可独立测试的功能点」。不满足就拆。

**归档判断：**
- 会在其他项目再遇到吗？ ✅ 是 — 任何项目的 Plan 都可能犯这个错
- 有明确的操作步骤吗？ ✅ 是 — 在 Plan 审查时检查 Task 粒度
- → **提炼为 Skill：在 `writing-plans` skill 中加强 Task 粒度检查规则**
