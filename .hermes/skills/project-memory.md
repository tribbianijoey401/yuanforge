# 项目记忆管理 Skill

> 定义 YuanForge 项目中知识记录的最佳实践。
> 项目记忆是项目的「大脑」——让任何 Agent 在任何时候都能理解项目上下文。

---

## 三份核心文档

| 文档 | 用途 | 更新时机 |
|------|------|---------|
| `ARCHITECTURE.md` | 系统架构概览 | 架构变更时 |
| `DECISIONS.md` | 技术决策日志 (ADR) | 做技术选型时 |
| `GLOSSARY.md` | 项目术语表 | 引入新概念时 |

---

## ARCHITECTURE.md — 架构决策记录

### 内容结构

```markdown
# 架构决策记录

## 项目概述
- 项目名称、目标、核心功能
- 技术栈总览

## 系统架构
- 架构图（ASCII art 或 Mermaid）
- 模块划分和职责
- 数据流

## 关键设计模式
- 使用的设计模式及其原因

## 外部依赖
- 数据库、缓存、消息队列等
- 第三方服务

## 部署架构
- 环境配置
- 容器化方案
```

### 更新时机

- ✅ 新增模块/服务
- ✅ 修改数据流
- ✅ 引入新的外部依赖
- ✅ 架构重构
- ❌ 小修小补（不需要）

## DECISIONS.md — 技术决策日志 (ADR)

### ADR 格式

每条决策记录包含：

```markdown
## YYYY-MM-DD: 决策标题

**状态：** 提议 / 已采纳 / 已废弃

**背景：** 为什么需要做这个决策？

**决策：** 我们选择了什么？

**备选方案：**
- 方案 A：优点 / 缺点
- 方案 B：优点 / 缺点

**后果：** 这个决策带来的影响
```

### 示例

```markdown
## 2026-06-27: 选择 FastAPI 作为后端框架

**状态：** 已采纳

**背景：** 需要构建 REST API 服务，要求高性能和自动文档生成。

**决策：** 选择 FastAPI

**备选方案：**
- FastAPI：异步支持好，自动 OpenAPI 文档，类型安全 — 社区相对 Flask 小
- Flask：简单灵活，社区大 — 缺少异步支持和自动文档
- Django REST：功能全 — 太重，不适合微服务

**后果：** 需要用 Python 3.11+，团队需要学习 async/await
```

### 更新时机

- ✅ 选择技术栈/框架
- ✅ 选择数据库/存储方案
- ✅ 选择架构模式
- ✅ 废弃旧的决策
- ❌ 不需要（日常开发不需要 ADR）

## GLOSSARY.md — 项目术语表

### 格式

```markdown
# 项目术语表

## 业务术语
- **用户 (User)：** 注册并登录系统的使用者
- **会话 (Session)：** 用户的一次登录周期

## 技术术语
- **Token：** JWT 认证令牌
- **Worker：** 后台任务处理器

## 缩写
- **CRUD：** Create, Read, Update, Delete
- **JWT：** JSON Web Token
```

### 更新时机

- ✅ 引入新的业务概念
- ✅ 使用新的技术术语
- ✅ 定义项目特定的缩写
- ❌ 不需要（通用术语不收录）

## 与实践的整合

### Phase 1（架构阶段）
- 架构师产出技术选型 → 写入 `DECISIONS.md`
- 架构设计 → 写入 `ARCHITECTURE.md`

### Phase 3（收尾阶段）
- 检查是否有遗漏的决策 → 补写 ADR
- 新增术语 → 加入 `GLOSSARY.md`

### 日常开发
- Agent 在开始工作时先读这三份文档
- 发现文档与实现不一致 → 标记为 issue 或直接修复

## Hermes Memory 的使用

对于跨会话的持久偏好和环境信息，使用 Hermes 的 `memory` 工具（而非这三份文档）：

| 存什么 | 工具 |
|--------|------|
| 架构决策 | → `DECISIONS.md` |
| 项目术语 | → `GLOSSARY.md` |
| 用户偏好 | → `memory` 工具 |
| 环境配置 | → `memory` 工具 |
| 工具经验/技巧 | → `skill` 工具 |
